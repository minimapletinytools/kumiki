- To create a "butt" variant of a corner joint (where one timber continues through without an end cut), swap from `CornerJointTimberArrangement` to `ButtJointTimberArrangement`, map timber1→butt_timber and timber2→receiving_timber, and simply omit the fork/receiving timber's end-cut computation. The rest of the geometry (tongue cheek removal, fork slot, tongue end cut) is identical.
- For butt-variant joints where only one timber is end-cut, CSG slot prisms in the non-end-cut timber must over-extend past the computed depth by at least the timber's max cross-section dimension. Without over-extension, the slot is clipped to the centerline depth and doesn't accommodate the angled tongue tip at non-90° joint angles. The overshoot is harmless because it falls outside the timber body.
- For plane-aligned corner joints that are not strictly 90° (e.g., tongue-and-fork corner joint), validate with `CornerJointTimberArrangement.check_plane_aligned()` and separately assert timbers are not parallel.

- **CRITICAL: OCP HalfSpace approximation direction** — When approximating a `HalfSpace(normal, offset)` as a large box, the box must extend only in the +normal direction from the plane (i.e., from `offset/|n|` to `offset/|n| + extent`), NOT symmetrically in both directions. If the box extends both ways, it covers the entire space and `BRepAlgoAPI_Cut` will produce an empty (void) shape, causing timbers to vanish. CadQuery's boolean kernel is more tolerant of overlapping extents, but OCP's is not. The HalfSpace convention is: the kept region is `{P : P·normal >= offset}`.
- **cadquery/OCP STEP export**: cadquery's `transformShape(cq.Matrix(...))` internally converts via `gp_GTrsf` → `gp_Trsf` which has extremely tight orthogonality tolerances and rejects even perfectly orthogonal matrices. Fix: use OCP's `gp_Trsf` directly with `gp_Trsf.SetValues()` + `wp.val().moved(cq.Location(trsf))`. Note: `.moved()` (lowercase, cadquery Shape method) not `.Moved()` (OCP method — not available on cadquery's Solid wrapper).
- **cadquery boolean ops can produce empty solids**: When a Difference fully subtracts a solid, cadquery returns a `Compound` with 0 solids. `_union_to_cq` and `_difference_to_cq` must skip empty children/subtracts (check `wp.solids().size() > 0`) to avoid "no solid on the stack to union" errors.
- **cadquery Workplane stack management**: After applying transforms via `.moved()`, wrap the result with `cq.Workplane("XY").newObject([moved_shape])` to create a proper Workplane with the solid on the stack. Using `cq.Workplane().add(shape)` doesn't properly set up the stack for subsequent boolean operations.
- **STEP units are millimetres**: Kumiki uses metres internally, but cadquery/OCP/STEP all use millimetres. All linear dimensions and positions must be multiplied by 1000 (`_to_mm` helper) when creating cadquery geometry. Direction vectors (unit vectors, rotation matrices) must NOT be scaled.

- **Math architecture in rule.py**: All numeric evaluation is bottlenecked through `giraffe_evalf(expr)` (using `GIRAFFE_EVALF_PRECISION=10`). Math operations use a three-layer pattern: `giraffe_*(…, collapse_mode)` base → `safe_*()` (smart collapse, backwards-compatible) → `numeric_*()` (always collapse to Float). `CollapseMode.SMART` collapses only when `is_complex_expr(expr, max_nodes=30)` detects complexity; `ALWAYS` eagerly collapses; `NEVER` preserves symbolic form. The old timeout/signal/lambdify/freeze_constants hacks have been removed.
- **normalize_vector → safe_normalize_vector**: The canonical name is now `safe_normalize_vector` (and `safe_magnitude` instead of `vector_magnitude`). The old names exist as deprecated wrappers. New code should use the `safe_*` or `numeric_*` names.
- **When to use numeric_* vs safe_***: Use `numeric_*` in hot paths where symbolic form is unnecessary (e.g., CSG operations in cutcsg.py). Use `safe_*` when results feed into further geometry construction (e.g., joints, construction helpers) where preserving exact Rationals matters.
- **Never use raw .evalf()**: All `.evalf()` calls must go through `giraffe_evalf()` so the evaluation strategy can be changed globally (e.g., switch to lambdify, add caching/timeouts) without touching callers.
- **`safe_compare` now takes two expression args**: `safe_compare(a, b, Comparison.GE)` — compares `a` against `b`. The old 2-arg form `safe_compare(expr, Comparison.XX)` (implicit zero) is gone. Use `Comparison.GE` / `Comparison.GT` enum members (not `GTE`).
- **`safe_zero_test` / `safe_equality_test` replace `zero_test` / `equality_test`**: `safe_zero_test(expr)` calls `safe_compare(expr, 0, Comparison.EQ)`. `safe_equality_test(a, b)` calls `safe_compare(a, b, Comparison.EQ)`. The old names exist as deprecated wrappers. `fast_zero_test` and `fast_equality_test` have been removed.
- **`adopt_csg` uses `numeric_*` internally**: The CSG coordinate transformation in `adopt_csg` (cutcsg.py) uses `numeric_transform_vector` and `numeric_dot_product` instead of `safe_*`, since CSG ultimately becomes triangles. This means CSG boundaries after `adopt_csg` are Float-valued, not exact Rationals. Tests that check `contains_point` near exact boundaries (e.g., z=5 exactly) should use points slightly off the boundary (e.g., z=5.1) to avoid Float precision edge cases.
- **Face comparison uses `numeric_dot_product`**: `_get_closest_oriented_face_from_faces` in timber.py uses `numeric_dot_product` instead of SymPy's `.dot()` method. This is safe because face selection only needs approximate alignment, and the speedup is significant (eliminates ~3600 slow SymPy matrix dot calls).
- **Timber alignment checks use `numeric_dot_product`**: `are_timbers_parallel`, `are_timbers_orthogonal`, `are_timbers_face_aligned`, `are_timbers_plane_aligned` in timber_shavings.py all use `numeric_dot_product`. These are boolean classification checks that don't need symbolic exactness.
